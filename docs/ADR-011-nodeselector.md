# ADR-010: 워크로드별 노드풀 분리 및 노드 셀렉터 적용

## 상태

**채택 (Accepted)**

## 맥락 (Context)

-   현재 모든 워크로드가 `default-pool`이라는 단일 노드풀에서 실행되고 있어 리소스 경합 및 장애 전파의 위험이 있습니다.
-   API, 백그라운드 작업, 운영 도구 등 성격이 다른 워크로드들은 요구하는 리소스 사양(CPU, Memory, IO)과 스케일링 정책이 다릅니다.
-   특정 워크로드의 리소스 과다 사용이 다른 워크로드의 성능에 영향을 주거나, 노드 장애 발생 시 중요 워크로드의 가용성을 보장하기 어렵습니다.

## 결정 (Decision)

**워크로드 특성에 따라 노드풀을 분리하고, `nodeSelector`를 사용하여 각 Pod가 지정된 노드풀에 스케줄링되도록 강제합니다.**

-   **`api-pool`**: 외부 사용자 트래픽을 처리하는 API 서버를 위한 노드풀. CPU 중심의 머신 타입을 사용하고, 트래픽에 따라 빠르게 스케일링되도록 설정합니다.
-   **`worker-pool`**: 메시지 큐 처리, 데이터 배치 작업 등 백그라운드 워크로드를 위한 노드풀. Memory 중심의 머신 타입을 사용합니다.
-   **`ops-pool`**: ArgoCD, Prometheus, Grafana 등 운영/모니터링 도구를 위한 노드풀. 안정적인 운영을 위해 별도로 격리합니다.
-   각 애플리케이션의 Kubernetes Deployment/Rollout 설정에 `nodeSelector` 필드를 추가하여 `workload-type: api`, `workload-type: worker`, `workload-type: ops`와 같은 레이블을 통해 적절한 노드풀을 지정합니다.

## 이유 (Rationale)

-   **안정성 및 격리**: 워크로드별 리소스 사용을 격리하여 한 워크로드의 문제가 다른 워크로드로 전파되는 것을 방지합니다. (e.g., worker의 높은 CPU 사용량이 api 성능에 영향을 주지 않음)
-   **비용 효율성**: 각 워크로드의 특성에 맞는 최적의 머신 타입을 선택하여 불필요한 리소스 낭비를 줄이고 비용을 최적화할 수 있습니다.
-   **유연한 스케일링**: 워크로드별로 독립적인 오토스케일링 정책을 적용하여, 트래픽이나 작업량 변화에 더 유연하고 빠르게 대응할 수 있습니다.
-   **가시성 및 관리 용이성**: 클러스터의 리소스 사용 현황을 워크로드별로 명확하게 파악하고 관리하기 용이해집니다.
