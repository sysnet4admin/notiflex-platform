# ADR-015: Kubernetes CronJob을 이용한 주기적인 헬스체크 구현

## 상태

**채택 (Accepted)**

## 맥락 (Context)

-   운영 중인 서비스(e.g., notiflex-api)나 외부 연동 서비스의 상태를 주기적으로 확인하여 장애를 조기에 감지하고 싶습니다.
-   예를 들어, 특정 API 엔드포인트가 정상적으로 응답하는지, 데이터베이스 연결이 유효한지 등을 정기적으로 검사할 필요가 있습니다.
-   이를 통해 잠재적인 문제를 사용자에게 영향을 미치기 전에 파악하고 대응할 수 있습니다.

## 결정 (Decision)

**Kubernetes `CronJob` 리소스를 사용하여 특정 시간 간격으로 헬스체크 스크립트를 실행하는 Pod를 생성합니다.**

-   헬스체크 로직을 담은 간단한 컨테이너 이미지를 만듭니다. (e.g., `curl`이나 `grpc-health-probe`와 같은 도구를 사용하는 스크립트)
-   `CronJob` 매니페스트를 작성하여, 이 컨테이너를 정해진 스케줄(e.g., 매 5분마다)에 따라 실행하도록 정의합니다.
-   헬스체크 작업(Job)은 안정적인 실행 환경을 보장하고 다른 워크로드에 영향을 주지 않도록 `ops-pool`에서 실행되도록 `nodeSelector`를 설정합니다.
-   헬스체크 실패 시, CronJob의 로그나 이벤트, 또는 Prometheus 알림(향후 연동)을 통해 운영팀에 통지합니다.

## 이유 (Rationale)

-   **Kubernetes 네이티브 접근**: `CronJob`은 Kubernetes가 기본적으로 제공하는 기능으로, 외부 스케줄링 도구나 라이브러리(e.g., Celery, Quartz)를 도입할 필요 없이 간단한 주기적 작업을 구현할 수 있습니다. 이는 시스템의 복잡성을 낮추고 관리 부담을 줄여줍니다.
-   **경량성 및 효율성**: 단순히 주기적으로 스크립트를 실행하는 요구사항에 대해, 별도의 워크플로우 엔진이나 데몬을 항상 띄워두는 것보다 `CronJob` 방식이 훨씬 가볍고 리소스 효율적입니다. Job이 실행될 때만 Pod가 생성되고 끝나면 사라집니다.
-   **워크로드 격리**: 헬스체크와 같은 관리/운영 목적의 작업을 `ops-pool`에서 실행함으로써, 실제 사용자 트래픽을 처리하는 `api-pool`의 리소스를 방해하지 않고 안정적으로 작업을 수행할 수 있습니다.
-   **선언적 관리**: CronJob의 스케줄, 실행할 컨테이너 이미지, 재시도 정책 등을 모두 YAML 매니페스트로 선언적으로 관리할 수 있어, GitOps 워크플로우에 쉽게 통합됩니다.
